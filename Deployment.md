# 企业信息化平台物理部署与资源配置容量规划

## 1. 概述与核心原则

在构建企业底层基础设施时，无论是选择公有云（ECS）还是本地虚拟化环境，资源配置的精细化直接关系到事务吞吐量（TPS）、响应延迟以及IT成本。
* **过度配置（Overprovisioning）**：不仅浪费预算，还可能在虚拟化环境中因 CPU 就绪时间（CPU ready time）延迟而拖慢性能。
* **资源不足（Underprovisioning）**：直接导致高峰期的服务拒绝或瘫痪。

本指南提供**不依赖具体编程语言或框架**（如 PHP/Java/Node）的通用型资源配置基准模型。

## 2. 核心节点的资源配置规范

### 2.1 接入层与 Web 服务器 (Access & Web Server)
* **业务特征**：处理海量并发 TCP 连接、SSL/TLS 卸载、静态资源代理转发。网络收发包极高，CPU中断频繁，内存容量要求相对较小。
* **配置基准**：
  * **实例类型**：计算优化型（如 C 系列）或高主频通用型。
  * **vCPU 与内存配比**：**1:1 或 1:2**。
  * **单节点基准**：`4核 8GB` 或 `8核 16GB`。
* **部署建议**：配合应用负载均衡器（ALB/SLB）结合弹性伸缩组（Auto Scaling）进行横向扩展（Scale-out），通过 NUMA 架构对齐技术提供极低的网络包转发延迟（PPS）。

### 2.2 核心应用与中间件服务器 (Core App & Middleware)
* **业务特征**：承载复杂的业务逻辑计算、数据组装与服务间远程过程调用（RPC）。大部分现代企业级运行环境（如 JVM, V8, CLR）需要较大堆内存及预留垃圾回收（GC）CPU 周期。
* **配置基准**：
  * **实例类型**：通用型实例族。
  * **vCPU 与内存配比**：严格控制在 **1:4**。
  * **单节点基准**：`8核 32GB` 或 `16核 64GB`。
* **容量测算**：
  * 针对重度事务型 ERP/CRM，每处理 **2000-6000 个事务行**分配 1 个专用 vCPU。
  * **每个核心**可支持 **10-15 名**企业重度并发用户，或 25-50 名普通查询用户。
* **部署建议**：对于大型批处理作业（Batch Processing），建议与在线交互负载在物理/虚拟机组别上分离，防止资源抢占。

### 2.3 数据库服务器 (Relational Database)
* **业务特征**：事务型（OLTP）负载。性能瓶颈为磁盘随机读写速率（IOPS）与事务重做日志（Redo Log）落盘延迟。引擎需要庞大的缓冲池（Buffer Pool）常驻内存。
* **配置基准**：
  * **实例类型**：存储优化型（如 i 系列），必须挂载 **本地 NVMe SSD** 或 **极速型云盘**。
  * **vCPU 与内存配比**：**不低于 1:4**（大型数仓建议 1:8）。
  * **容量测算**：处理 **3000-15000 个事务行**需 1 个计算核心，每核心配备 2GB-16GB 内存。

### 2.4 分布式缓存与内存数据库 (Distributed Cache)
* **业务特征**：极端内存密集型负载。所有读写操作在系统内存中完成。
* **配置基准**：
  * **实例类型**：内存优化型（如 r 系列）。
  * **vCPU 与内存配比**：**高达 1:8**。
* **部署建议**：最大化单节点内存容量以控制 CPU 核心软件授权成本。

## 4. 物理部署的高可用容灾原则
在完成资源量化之后，硬件实例在公有云的映射需遵循跨域容灾原则：

* **应用负载均衡 (ALB)** 必须跨越至少两个可用区 (AZ)。
* **Kubernetes/ECS Worker 节点** 必须打散分布在同一 VPC 内的多个物理机房。
* **数据集群** 确保主节点与读写副本位于完全独立的故障域，并开启实时 Binlog/日志同步接管机制。
